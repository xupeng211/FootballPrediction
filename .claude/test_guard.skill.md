# Test Guard Skill - 测试保护约束

**约束等级**: 🔴 RED（必须遵守）

## 核心原则
- 测试是保护网，不是负担
- 优先修复源码，而非放宽测试
- 每个测试必须有明确的业务价值

## 测试失败处理准则

### 1. 优先级排序
1. **修复源码** (首选)
   - 保持测试断言不变
   - 修复导致测试失败的源码问题
   - 确保测试真正反映业务需求

2. **修正测试** (备选)
   - 仅当测试确实不合理时
   - 必须提供充分理由
   - 新测试必须更严格地覆盖真实行为

3. **放宽断言** (禁止)
   - 绝对禁止为了通过测试而放宽断言
   - 禁止将具体值改为范围
   - 禁止删除重要断言

### 2. 禁止的断言弱化

#### ❌ 弱断言列表
```python
# 禁止使用
assert True  # 无意义
assert not None  # 过于宽泛
assert result is not None  # 应该检查具体内容
assert len(result) > 0  # 应该检查具体长度
assert response.status_code == 200  # 应该同时检查响应内容
```

#### ✅ 强断言示例
```python
# 推荐使用
assert prediction.home_win == 0.65
assert prediction.confidence > 0.6
assert "feature_x" in features.columns
assert all(0 <= p <= 1 for p in probabilities)
```

## 测试修改规范

### 修改测试前必须说明

```markdown
**测试文件**: [测试文件路径]
**原测试问题**: [为什么原测试不合理，具体说明]
**新测试价值**: [新测试覆盖了什么真实业务行为]
**风险评估**: [修改可能带来的风险]
```

### 常见的合理修改场景

1. **测试与业务需求不符**
   - 原测试: 断言返回 200
   - 问题: 未验证业务逻辑正确性
   - 修改: 增加业务结果断言

2. **测试过于脆弱**
   - 原测试: 断言精确的时间戳
   - 问题: 每次运行都失败
   - 修改: 使用时间范围或 mock

3. **测试覆盖不足**
   - 原测试: 只测试成功路径
   - 修改: 增加异常场景测试

### 禁止的修改场景

1. **为了通过而放宽**
   ```python
   # 禁止
   # 原测试: assert result["accuracy"] == 0.85
   # 修改后: assert result["accuracy"] > 0.5
   ```

2. **删除关键断言**
   ```python
   # 禁止
   # 删除: assert prediction.confidence > 0.7
   ```

3. **忽略异常测试**
   ```python
   # 禁止
   # 注释掉: assert raises(ValueError)
   ```

## 特定测试类型规则

### 单元测试
- 每个测试只验证一个行为
- 使用描述性测试名称
- 包含边界条件测试

### 集成测试
- 测试组件间交互
- 验证数据流正确性
- 包含错误恢复测试

### 性能测试
- 设置明确的性能基准
- 包含资源使用监控
- 避免依赖硬件性能

## 测试质量检查清单

### 新增测试时检查
- [ ] 测试名称清晰描述测试内容
- [ ] 使用 Arrange-Act-Assert 模式
- [ ] 断言具体而有意义
- [ ] 包含边界情况
- [ ] 不依赖外部系统状态

### 修改测试时检查
- [ ] 已说明修改理由
- [ ] 新测试更严格而非宽松
- [ ] 保持测试独立性
- [ ] 没有降低测试覆盖率

## 违规处理

### 自动检测
1. 发现弱断言 → 立即警告
2. 发现删除断言 → 阻止执行
3. 发现无理由放宽 → 要求说明

### 处理流程
1. ⚠️ 输出警告: "测试质量风险"
2. 📝 要求提供修改理由
3. ✅ 或人工确认后才继续
4. 📊 记录所有测试修改

## 最佳实践

### 编写可维护的测试
1. 使用测试工厂函数
2. 创建有意义的测试数据
3. 保持测试简单专注

### 测试命名规范
```python
def test_predict_match_returns_valid_probability_distribution():
def test_prediction_confidence_below_threshold_raises_error():
def test_model_training_with_insufficient_data_fails():
```

### 测试数据管理
1. 使用 fixture 管理测试数据
2. 避免硬编码测试值
3. 清理测试产生的数据

记住：**好的测试是项目的保护网，不是开发的障碍**
