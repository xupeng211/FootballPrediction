# Change Impact Skill - 变更影响分析

**约束等级**: 🔴 RED（必须遵守）

## 核心原则
- 每次修改前必须评估影响
- 影响不可评估时必须明确标示
- 未知影响等于高风险

## 影响层级定义

### 🟢 轻量级影响
- **特征**: 单文件修改，无依赖变化
- **示例**: 修复计算错误、优化注释
- **测试要求**: 单元测试 + 基本集成测试

### 🟡 中级影响
- **特征**: 多文件修改，局部依赖变化
- **示例**: 增加新特征、修改配置项
- **测试要求**: 完整单元测试 + 相关集成测试

### 🔴 重大影响
- **特征**: 跨模块修改，架构变化
- **示例**: 修改核心算法、变更数据结构
- **测试要求**: 全面回归测试 + 性能测试

## 自动化影响检测

### 静态分析检查
1. **依赖图分析**
   - 查找所有依赖当前模块的代码
   - 生成调用链图
   - 标记循环依赖

2. **接口变更检测**
   - 方法签名变化
   - 参数类型变化
   - 返回值变化
   - 异常声明变化

3. **配置变更影响**
   - 配置项增删改
   - 环境变量依赖
   - 默认值变化

### 动态分析建议
1. **运行时影响**
   - 热点代码路径
   - 性能关键路径
   - 并发访问点

2. **测试覆盖分析**
   - 当前测试覆盖率
   - 未覆盖的执行路径
   - 需要补充的测试

## 风险等级判定

### 🟢 低风险（可自行决策）
- 修改纯工具函数
- 修改测试代码
- 更新文档注释
- 修复拼写错误

### 🟡 中风险（需要代码审查）
- 修改业务逻辑
- 增加新功能
- 优化性能
- 修改配置

### 🔴 高风险（需要架构师审批）
- 修改核心算法
- 变更数据结构
- 跨模块重构
- 修改公共接口

### ⚫ 极高风险（需要团队决策）
- 架构重大变更
- 技术栈切换
- 数据迁移
- 性能瓶颈改造

## 不确定性的处理

### 明确标识未知
```markdown
### ⚠️ 不确定性风险
以下影响无法完全评估：
- [未知影响1] - [无法评估的原因]
- [未知影响2] - [缺少的信息]

### 📝 需要进一步调研
- [调研项1] - [需要什么信息]
- [调研项2] - [谁可以提供帮助]

### 🛡️ 缓解措施
- 分阶段实施
- 灰度发布
- 监控告警
- 快速回滚方案
```

## 实施检查清单

### 修改前
- [ ] 完成影响分析报告
- [ ] 确定风险等级
- [ ] 制定测试计划
- [ ] 准备回滚方案

### 修改中
- [ ] 遵循最小修改原则
- [ ] 保持测试通过
- [ ] 记录实际影响
- [ ] 及时更新评估

### 修改后
- [ ] 执行回归测试
- [ ] 验证功能正确性
- [ ] 检查性能影响
- [ ] 更新文档

## 记住

**"影响不可评估" = "高风险"。任何不确定的修改都应该谨慎对待，宁可保守一些，也要确保系统稳定。**
