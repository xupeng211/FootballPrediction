## 保护级别定义

### 🔒 P0 - 严格冻结（Critical）
- **适用**: 预测核心、模型管理
- **规则**:
  - 禁止任何结构性修改
  - 仅允许紧急 Bug 修复
  - 需要架构师 + 技术负责人双签
  - 必须 100% 回归测试通过

### 🔒 P1 - 审批冻结（High）
- **适用**: 数据流水线、特征工程
- **规则**:
  - 需要架构师审批
  - 必须提供影响评估
  - 完整的集成测试
  - 性能基准对比

### 🔒 P2 - 测试冻结（Medium）
- **适用**: 高覆盖率工具类
- **规则**:
  - 必须保持测试覆盖率
  - 向后兼容性检查
  - 完整的单元测试

3. 重构架构设计
4. 变更依赖关系

#### ❌ 所有模块
1. 删除关键注释
2. 修改日志级别
3. 调整监控指标
4. 禁用安全检查

### 允许的操作

## 违规处理

### 自动检测机制
1. 文件修改检测
   - Git hooks 检查核心文件修改
   - 自动标记需要审批的 PR

2. 代码扫描
   - 检测接口签名变更
   - 监控核心逻辑修改
   - 识别测试覆盖率下降

3. 影响分析
   - 自动分析修改影响范围
   - 生成依赖关系图
   - 标记潜在风险点

### 处理流程
1. ⚠️ 检测到违规修改
2. 🚫 自动阻止提交
3. 📧 通知架构师和技术负责人
4. 📝 要求提供详细说明
5. ✅ 审批通过后才可继续
