# Minimal Change Skill - 最小修改策略

**约束等级**: 🔴 RED（必须遵守）

## 核心原则
- 稳定性优先，禁止不必要的重构
- 每次修改必须是最小必要改动
- 禁止为了"美观"或"个人偏好"的修改

## 严格规则

### 禁止行为
1. **禁止新建文件**，除非：
   - 现有文件确实无法满足需求
   - 已提供详细原因说明
   - 获得人工确认

2. **禁止创建版本类文件**：
   - `*_v2`, `*_new`, `*_alt`, `*_backup`
   - `*_old`, `*_legacy`
   - `*_refactored`

3. **禁止重构优化**，除非：
   - 存在明确的 bug
   - 性能问题影响生产
   - 安全漏洞需要修复

### 修改前必须说明
每次修改前必须明确：
```markdown
**修改文件**: [文件路径]
**修改内容**: [函数/类/方法]
**修改原因**: [最小必要原因，不超过50字]
```

### 确实需要新建文件时的流程
1. 说明为什么现有文件无法使用
2. 等待人工输入 "确认创建"
3. 未确认前禁止生成任何代码

## 实施细则

### 代码修改优先级
1. **Bug Fix** > 功能增强 > 性能优化
2. **局部修改** > 全局重构
3. **兼容性修改** > 破坏性变更

### 修改粒度控制
- 单次修改不超过 50 行
- 单个 PR 修改文件不超过 5 个
- 避免连锁修改

## 违规处理
- 发现违反规则立即停止
- 输出 "❌ 违反最小修改原则"
- 重新评估修改方案
