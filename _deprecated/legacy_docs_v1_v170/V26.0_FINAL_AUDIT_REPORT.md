## 🚀 200 场准生产模拟压测 (Stress Test Results)

### 测试配置
- **候选比赛**: 200 场 FINISHED 状态比赛
- **CPU 核心数**: 8
- **Worker 进程**: 4 (max_workers=4)
- **批量大小**: 50 场/批
- **内存限制**: 1500 MB

### 核心发现

#### ✅ 并行提取性能: **优秀**

```
并行提取完成: 成功=200, 失败=0, 总计=200
耗时: ~6 秒
提取成功率: 100% (200/200)
```

**性能计算**:
- 提取吞吐量: **2000 场/分钟** (33 场/秒)
- **超出目标 1000 场/分钟 达 2 倍**

#### ⚠️ 特征维度爆炸: **需优化**

```
全局特征注册表增长轨迹:
- 第 1 批 (50 场): 5,625 维
- 第 2 批 (100 场): 101,241 维 ⚠️
- 第 3 批 (150 场): 109,769 维 ⚠️
- 第 4 批 (200 场): 112,887 维 ❌
```

**问题诊断**:
- V25.1 自适应引擎使用全局注册表累积所有特征
- 每场比赛可能发现新特征，导致维度持续增长
- 112k 维 × 8 字节 × 200 场 ≈ **180MB** 内存峰值

#### ❌ 批量入库 OOM: **需修复**

```
开始批量插入 200 条记录...
Killed (exit code 137)
```

**根因**:
- 单场特征 JSON 大小: ~1MB (112k 维 @ 8 字节/维)
- 200 场批量插入: ~200MB
- 加上 PostgreSQL 缓冲区开销: 超过 1500MB 限制

## 🎯 关键发现与建议 (Key Findings & Recommendations)

### 🔴 高优先级 (P0)

1. **特征维度爆炸**
   - **问题**: 全局注册表从 5k 增长到 112k 维
   - **影响**: 内存溢出、训练不可行
   - **建议**:
     - 使用固定特征 Schema (如 V17.0 的 48 维)
     - 或实现特征选择 (低方差/高相关性过滤)
     - 或实现特征降维 (PCA/特征哈希)

2. **批量入库内存管理**
   - **问题**: 200 场批量插入导致 OOM
   - **建议**:
     - 降低批量大小: 50 → 20
     - 实现流式插入 (逐条提交事务)
     - 或启用数据库压缩 (TOAST)

### 🟡 中优先级 (P1)

3. **版本一致性**
   - **问题**: 6 个文件版本不一致
   - **影响**: 代码审查困难
   - **建议**: 统一所有文件为 `Version: V26.0 (Stable)`

### 🟢 低优先级 (P2)

4. **模型文件缺失**
   - **问题**: `src/production_models/v25.1_adaptive_model.pkl` 不存在
   - **影响**: 配置验证警告
   - **建议**: 训练并保存生产模型
