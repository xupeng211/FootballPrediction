### 2.1 Global L1 Scanner (赛程探测器)

**核心职责**: 从 OddsPortal 发现可采集的比赛，生成待提取任务队列。

#### 功能模块

| 模块 | 功能 | 输入 | 输出 |
|------|------|------|------|
| **URL 生成器** | 构建 OddsPortal 比赛列表 URL | 联赛 ID、日期范围 | URL 列表 |
| **列表解析器** | 从 HTML 中提取比赛基础信息 | HTML 响应 | match_id, oddsportal_url |
| **优先级计算器** | 根据比赛时间和热度计算优先级 | 比赛日期、联赛级别 | priority_score |

#### 关键配置

```python

    "priority_weights": {

        "ligue_1": 0.85,

        "serie_a": 0.8,
        "other": 0.5

```

#### 工作流程

```
1. [启动] 加载联赛配置 → 生成待扫描 URL 列表
2. [发现] 访问 OddsPortal → 解析比赛列表
3. [过滤] 排除已完成的比赛 → 排除已有数据的比赛
4. [排队] 写入 Redis Queue → 设置优先级分数
5. [报告] 记录发现的比赛数量 → Prometheus 指标
```

### 2.2 L2 Market Data Engine (多源提取引擎)

**核心职责**: 从单个 OddsPortal 比赛页面提取 5 个供应商的赔率数据。

#### 功能模块

| 模块 | 功能 | 关键技术 |
|------|------|----------|
| **内存提取器** | 从 DOM 中提取赔率数据 | Playwright Page.evaluate |
| **模糊匹配器** | 映射供应商名称到内部实体 | TheFuzz + 规则引擎 |
| **分布验证器** | 检测是否为标准 1X2 市场 | 统计检验 + 启发式规则 |
| **完整性审计器** | 计算 Score = 1/P1 + 1/P2 + 1/P3 | 数学校验 (1.02 < Score < 1.08) |

#### 内存提取逻辑

```javascript
// L2 内核提取算法 (JavaScript 注入)
const extractionLogic = () => {
    // 1. 提取页面全文
    const fullText = document.body.innerText;

    // 2. 正则匹配供应商 + 赔率模式
    const pattern = /([A-Za-z][A-Za-z0-9\s]{3,20})\s+(\d+\.\d{2})\s+(\d+\.\d{2})\s+(\d+\.\d{2})/g;

    // 3. 构建供应商 → 赔率 映射
    const vendorOdds = {};
    for (const match of fullText.matchAll(pattern)) {
        const [, vendor, h, d, a] = match;
        if (h >= 1.01 && h <= 50.00 && d >= 1.01 && d <= 50.00 && a >= 1.01 && a <= 50.00) {
            vendorOdds[vendor.trim()] = { h: parseFloat(h), d: parseFloat(d), a: parseFloat(a) };
        }
    }

    return vendorOdds;
};
```

#### 数学校验机制

```python
class IntegrityValidator:
    """完整性审计器 - 确保数据为标准 1X2 市场"""

    # 1X2 市场完整性范围
    MIN_INTEGRITY_SCORE = 1.02  # 最低阈值 (赔率偏高)
    MAX_INTEGRITY_SCORE = 1.08  # 最高阈值 (赔率偏低)
    IDEAL_SCORE = 1.05         # 理想值

    # 赔率值域
    MIN_ODDS = 1.01
    MAX_ODDS = 50.00

    @staticmethod
    def calculate_integrity(h: float, d: float, a: float) -> float:
        """计算完整性分数

        Formula: Score = 1/h + 1/d + 1/a

        理论值域:
        - 完美市场: ≈1.05
        - 高风险: <1.02 (赔率过高) 或 >1.08 (赔率过低)
        - 无效: =1.00 (不可能) 或 >1.10 (异常)
        """
        return 1.0/h + 1.0/d + 1.0/a

    @staticmethod
    def validate_distribution(h: float, d: float, a: float) -> tuple[bool, str]:
        """验证数据分布

        检查项:
        1. 赔率值域 [1.01, 50.00]
        2. 三个值互不相同
        3. 值域范围 > 0.5 (拒绝 all ~1.10)
        4. Draw 在合理区间
        """
        # 检查 1: 值域验证
        if not all(MIN_ODDS <= v <= MAX_ODDS for v in [h, d, a]):
            return False, "Odds outside valid range [1.01, 50.00]"

        # 检查 2: 互不相同
        if len({h, d, a}) != 3:
            return False, "Identical odds detected"

        # 检查 3: 范围验证
        if max(h, d, a) - min(h, d, a) < 0.5:
            return False, f"Odds too similar (range={max(h, d, a) - min(h, d, a):.3f})"

        # 检查 4: Draw 在合理区间 (启发式)
        sorted_odds = sorted([h, d, a])
        if not (sorted_odds[0] <= d <= sorted_odds[2]):
            return False, f"Draw {d} outside expected range"

        return True, "Valid 1X2 distribution"
```

#### 指数退避重试

```python
class ExponentialBackoffRetry:
    """指数退避重试策略"""

    BASE_DELAY = 2.0    # 基础延迟 2 秒
    MAX_ATTEMPTS = 3    # 最多重试 3 次

    @staticmethod
    def get_delay(attempt: int) -> float:
        """计算第 N 次重试的延迟时间

        Formula: delay = BASE_DELAY * 2^(attempt-1)

        Attempts | Delay (秒)
        ---------|------------
          1      |   2.0
          2      |   4.0
          3      |   8.0
        """
        return BASE_DELAY * (2 ** (attempt - 1))

    @staticmethod
    async def execute_with_retry(coro_factory, max_attempts: int = 3) -> object:
        """执行带指数退避的重试"""
        last_exception = None

        for attempt in range(1, max_attempts + 1):
            try:
                return await coro_factory()
            except Exception as e:
                last_exception = e
                if attempt < max_attempts:
                    delay = ExponentialBackoffRetry.get_delay(attempt)
                    logger.warning(
                        f"[Retry] Attempt {attempt}/{max_attempts} failed, "
                        f"retrying in {delay:.1f}s: {e}"
                    )
                    await asyncio.sleep(delay)

        raise last_exception if last_exception else Exception("All retry attempts failed")
```

### 4.2 为什么引入 Poetry 锁定环境？

#### 问题背景

传统的 `requirements.txt` 存在以下问题：

1. **版本不确定**：`pandas>=2.0.0` 在不同时间安装可能得到 `2.0.0` 或 `2.2.3`
2. **传递依赖不确定**：A 包依赖 B 包，B 包的版本可能变化
3. **安装顺序敏感**：包的安装顺序可能影响最终结果

#### Poetry 解决方案

```toml
# pyproject.toml (声明式)
[tool.poetry.dependencies]
pandas = "^2.0.0"  # 兼容性版本范围

# poetry.lock (确定性)
[[package]]
name = "pandas"
version = "2.2.3"  # 精确版本号

[[package.dependencies]]
name = "numpy"
version = "1.24.0"  # pandas 的精确依赖版本
```

#### 优势

```

├─────────────────────────────────────────────────────────────────────┤
│                                                                       │
```

### 4.3 为什么数据库设计必须满足 (match_id, source_name) 唯一约束？

#### 业务需求

1. **多供应商共存**：同一场比赛需要存储 5 个供应商的数据
2. **数据更新**：供应商数据可能在不同时间更新
3. **历史追踪**：需要保留数据的历史版本（可选）

#### 唯一约束设计

```sql
-- 表结构设计
CREATE TABLE metrics_multi_source_data (
    id BIGSERIAL PRIMARY KEY,
    match_id VARCHAR(255) NOT NULL,
    source_name VARCHAR(50) NOT NULL,

    -- 赔率数据
    init_h FLOAT,
    init_d FLOAT,
    init_a FLOAT,
    final_h FLOAT,
    final_d FLOAT,
    final_a FLOAT,
    integrity_score FLOAT,

    -- 验证元数据
    is_valid BOOLEAN DEFAULT FALSE,
    validation_error TEXT,

    -- 唯一约束 (关键设计)
    CONSTRAINT uq_match_source UNIQUE (match_id, source_name)
);

-- 创建索引
CREATE INDEX idx_match_id ON metrics_multi_source_data(match_id);
CREATE INDEX idx_source_name ON metrics_multi_source_data(source_name);
CREATE INDEX idx_uq_match_source ON metrics_multi_source_data(match_id, source_name);
```

#### Upsert 操作

```sql
-- 实现"一赛多行"的 Upsert
INSERT INTO metrics_multi_source_data (
    match_id, source_name,
    init_h, init_d, init_a,
    final_h, final_d, final_a,
    integrity_score, is_valid
) VALUES (
    '12345', 'Entity_P',
    2.05, 3.25, 3.60,
    2.03, 3.30, 3.65,
    1.0478, TRUE
) ON CONFLICT (match_id, source_name)
DO UPDATE SET
    init_h = EXCLUDED.init_h,
    final_h = EXCLUDED.final_h,
    integrity_score = EXCLUDED.integrity_score,
    is_valid = EXCLUDED.is_valid;
```

#### 设计优势
