## Why One Sample Is Not Trainable

The current local sample is not sufficient for real model training:

- It is a single scheduled match.
- It does not provide a finished-match label suitable for supervised learning.
- It cannot support train / validation / test separation.
- It cannot estimate class balance, calibration, league bias, draw bias, or temporal drift.
- It cannot validate leakage controls.
- The existing training path requires a finished historical dataset and writes model artifacts, so it remains blocked.

## Multisample Data Strategy

Candidate source tables:

- `matches`: schedule, teams, kickoff time, status, final scores, and `actual_result` labels.
- `bookmaker_odds_history`: bookmaker snapshots, market type, odds movement, and collection timestamps.
- `raw_match_data`: source payloads for feature extraction and audit traceability.
- `l3_features`: curated feature JSONs produced by authorized L3 gates.
- `match_features_training`: model-ready feature rows and adaptive feature payloads.
- `predictions`: model output history for later monitoring only, not for training labels.
- Future historical samples: finished matches collected under an explicit dataset runbook.

Minimum dataset readiness criteria:

- At least several hundred finished matches before training is considered.
- Every training row must have a reliable `actual_result` or final score.
- Scheduled matches must be excluded from label generation.
- Each feature row must be tied to kickoff-time availability rules.
- Odds and feature snapshots must carry usable `collected_at` / `computed_at` timing.
- Samples with missing labels, post-match-only features, or ambiguous time ordering must be excluded or explicitly quarantined.

## Label Strategy

Primary label:

- 3-class result: `home_win`, `draw`, `away_win`.

Label source:

- Prefer `matches.actual_result` if it is populated and consistent with final score.
- Otherwise derive from `home_score` and `away_score` only for finished matches.

Explicit exclusions:

- Scheduled matches cannot create labels.
- Cancelled, postponed, abandoned, or status-ambiguous matches require a separate policy before use.
- Score regression, goals totals, corners, cards, and bet-specific labels are out of scope for the first real model preflight.

## Feature Strategy

Allowed feature sources for first training preflight:

- `match_features_training` required identity fields and nullable numeric fields.
- `match_features_training.adaptive_features` only after a versioned schema is documented.
- L3-derived summaries that are demonstrably available before kickoff.
- Odds features from `bookmaker_odds_history` using the latest snapshot before kickoff or a clearly defined cutoff.

Restricted feature sources:

- Raw payload fields that include post-match statistics, final scores, shotmaps after kickoff, or status updates after the match.
- Any feature with `computed_at` after kickoff unless it is explicitly labeled as unavailable for pre-match prediction.
- `predictions` rows as features for training the first model.
- ELO values that were not generated by an authorized, time-aware ELO process.

Missing data rules:

- Missing `elo_features` must remain missing; do not fabricate or backfill without a dedicated ELO phase.
- Missing odds snapshots must be encoded as missing, not replaced with arbitrary defaults.
- Missing feature families should be measured and reported before training.

## Leakage Controls

Hard rules:

- No future match outcomes in features.
- No post-kickoff or post-final-whistle statistics in pre-match training rows.
- No random cross-time split that mixes future seasons into training for past validation.
- No use of prediction rows to train the model that produced those predictions.
- No feature computation that sees validation / test labels while preparing training features.

Timing checks required before training:

- `feature_time <= cutoff_time <= kickoff_time` for pre-match prediction features.
- Odds snapshot time must be earlier than the cutoff policy.
- L3 `computed_at` must be evaluated against feature availability, not blindly trusted.
- Any backfilled historical feature must preserve the original observation time or be excluded.

## Split Strategy

Recommended split:

- Chronological split only.
- Train on older completed matches.
- Validate on the next time window.
- Test on the most recent held-out time window.

Initial policy proposal:

- Train window: oldest 70 percent by kickoff date.
- Validation window: next 15 percent by kickoff date.
- Test window: latest 15 percent by kickoff date.

Additional requirements:

- Report class distribution in each split.
- Report league and season coverage in each split.
- Avoid splitting the same competition week across train and validation when leakage risk is high.
- Keep a final untouched test set for model approval.

## Model Artifact Strategy

Artifact storage:

- Real model binaries must not be committed to Git.
- Local ignored paths should remain `models/` and `model_zoo/`.
- `docs/MODEL_ARTIFACTS.md` and `config/model_artifacts.example.json` are the current scaffold.

Required artifact metadata:

- `model_version`
- `training_data_hash`
- `train_window`
- `validation_window`
- `test_window`
- `feature_schema_version`
- `label_policy_version`
- `created_at`
- `code_commit`
- `metrics`
- `calibration_summary`
- `artifact_path`
- checksum

Before authorization:

- Do not load production model artifacts.
- Do not write model artifacts.
- Do not run training scripts.
- Do not run prediction scripts.

## Evaluation Strategy

Required metrics:

- Accuracy.
- Log loss.
- Brier score.
- Calibration curve or calibration bins.
- Class distribution and per-class recall.
- Confusion matrix.
- Baseline comparison against home-prior, market-prior, and uniform / draw-aware baselines.
- Draw bias and home bias checks.

Approval criteria must be defined before training:

- Minimum sample count.
- Maximum tolerated missingness.
- Minimum calibration quality.
- Required out-of-time validation performance.
- Required comparison against baseline.

## Prediction Write Strategy

Real prediction writes must require:

- An approved model artifact.
- A model artifact manifest and checksum.
- A feature schema version compatible with the artifact.
- A target match scope and cutoff policy.
- A no-write preview that does not mutate DB state.
- A separate explicit authorization for `predictions` writes.

Manual / baseline predictions:

- Must be clearly marked by model version, such as `P4_LOCAL_BASELINE`.
- Must not be described as model outputs.
- Must not be mixed into model evaluation as real inference results.
- Must remain local pipeline closure evidence only.

Dry-run previews:

- Must not be treated as real prediction execution.
- Must not load model artifacts unless a future phase explicitly authorizes an artifact validation path.

Risk findings:

- Training scripts can train models and write artifacts.
- Prediction scripts can enter model loading / inference code.
- Existing `predict:dry` is not considered a Phase-safe no-op because it still enters prediction pipeline code.
- Model artifact documentation exists, but production artifact storage and approval policy remain scaffold-level.
- No approved `make data-dataset-status` or `make data-training-dataset-dry-run` gate exists yet.
